# Core release journey audit - 0.0.40

This report cross-checks Core-applicable rows in
[`RELEASE_TEST_CHECKLIST.csv`](RELEASE_TEST_CHECKLIST.csv) against
[`MANUAL_RELEASE_TESTS_0.0.40.md`](MANUAL_RELEASE_TESTS_0.0.40.md) and the strict classifications in
[`P0_P2_INTEGRATION_COVERAGE.md`](P0_P2_INTEGRATION_COVERAGE.md).

## Scope and calibration

- Core scope: 107 journeys - 51 P0, 48 P1, 8 P2, and 0 P3.
- Excluded as Pro-only: #2, #83-104, #107-128, and #138-140.
- Rows that combine Core and Pro outcomes remain in scope, but this report names only the Core
  portion. Pro-only clauses must be marked N/A during a Core pass.
- `High` means the decisive product seam is exercised without replacing Off Grid code. The manual
  pass still checks the installed artifact, operating-system boundary, or pixels.
- `Medium` means material production owners run, but a native, network, installer, renderer, or
  relaunch boundary remains split or controlled.
- `Low` means the decisive boundary is still manual, the current UI test substitutes an Off Grid
  preload/service, or the manual checklist does not explicitly represent the CSV journey.

## Manual-checklist gaps

- #15 and #16 are represented only by the Pro-labelled permission-recovery item, even though Core
  chat owns microphone recording and its System Settings recovery path. Add an explicit Core
  microphone grant/deny/retry check.
- #40 queued message order has automation but no explicit manual step. Add a two-message queue check
  while the first response is still pending.
- #49 long-answer cutoff has automation but no explicit manual step. Add a configured-cap check that
  verifies visible cutoff state and restoration after reopening the conversation.
- #50 desktop Back/Forward shortcuts have automation but no explicit manual step. Add Cmd+[ and Cmd+]
  across a project-scoped screen and another route.
- #141 development override behavior has automation but no explicit manual check. Record it as a
  development-only Core gate, separate from the packaged entitlement check.
- #31 and #59 are only indirectly represented by the generic large-collection/resize checks. Name
  Models density and Projects master-detail/grid explicitly so a generic resize pass cannot skip
  either surface.
