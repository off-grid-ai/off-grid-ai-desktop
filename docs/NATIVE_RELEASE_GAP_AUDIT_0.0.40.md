# Off Grid AI Desktop 0.0.40 native and release gap audit

This audit supplements [`RELEASE_TEST_CHECKLIST.csv`](RELEASE_TEST_CHECKLIST.csv) and
[`MANUAL_RELEASE_TESTS_0.0.40.md`](MANUAL_RELEASE_TESTS_0.0.40.md). It covers native, release, and
cross-cutting journeys whose important boundary is not fully represented in the current 155-row
release checklist.

The rows below are proposals for a future checklist revision. They are not pass results. A green
automated seam never replaces the remaining operating-system, hardware, provider, network, or
final-artifact boundary named by the row.

## Confidence scale

- **High** - repository code, tests, workflows, or release documents directly show the gap.
- **Medium** - the OS or hardware boundary is clear, but repository evidence is narrower.
- **Low** - a useful exploratory release journey with limited direct repository evidence.

Confidence describes confidence that the checklist gap exists. It does not describe confidence
that the current build passes the journey.

## Critical findings

### 1. The gateway's network boundary contradicts its documentation

- `src/main/model-server.ts:929` says the gateway is bound to loopback.
- `src/main/api-docs.ts:120-126` says no API key is required because the gateway listens only on
  `127.0.0.1`.
- `src/main/model-server.ts:1209-1213` actually binds to `0.0.0.0` and logs that it is reachable on
  the LAN.
- `src/main/model-server.ts:1086-1096` applies a loopback check only to `POST /v1/settings`. The
  broader gateway route surface does not have an equivalent access-control seam.
- Existing gateway integration tests use loopback. No LAN reachability, authentication, or
  cross-interface test was found.

This must be resolved as a release policy, not treated only as a copy defect. If LAN access is
intentional, the release needs explicit authentication, disclosure, threat-boundary, and firewall
coverage. If it is not intentional, the listener must be constrained to loopback.

### 2. OAuth callback state matching accepts the sole pending request as a fallback

`src/main/mcp-oauth.ts:176-185` reads the callback state, but resolves the callback with:

```ts
pendingByState.get(state) ??
  (pendingByState.size === 1 ? [...pendingByState.values()][0] : undefined)
```

Therefore, an absent or mismatched `state` can complete the only pending OAuth flow. No automated
callback-state, replay, concurrent-flow, timeout, or callback-port-collision test was found. The
Google client test only verifies shared OAuth client configuration.

The release journey must prove that only an exact pending state can receive a token and that a
state cannot be reused.

### 3. Release sources disagree about artifact and platform topology

- `.github/workflows/release.yml:25-29` defines one signed and notarized Pro-capable DMG whose Pro
  features unlock in place.
- `MANUAL_RELEASE_TESTS_0.0.40.md:3-8` also distinguishes the single production artifact from local
  Core and Pro diagnostic variants.
- The existing CSV and `RELEASE_DESKTOP.md` still contain language that can be read as separate
  production Core and Pro artifacts.
- `.github/workflows/release.yml:25` says "macOS-only for now", while lines 30-32 describe a Windows
  build published into the same release.

The release team must establish one source of truth for the production artifact, locked and
entitled test states, filenames, bundle identifiers, update feeds, and whether Windows is shipping.

## Audit conclusion

The current manual release document is stronger than the CSV for trust, detached installation,
and applied updates. It does not close the gateway exposure, OAuth callback state, active TCC
revocation, sleep and lock, multi-display privacy, packet-level egress, real device churn, or
process-death boundaries.

NR-01, NR-12, NR-13, and NR-17 require a release or implementation decision before they can be
recorded as ordinary manual checks. The other rows can be added as supplemental release journeys
with their current automation status and remaining boundary preserved.
